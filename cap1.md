## 基本信息
| 能力域定位 | 设计和管理跨 8+ 国家/地区的全球网络拓扑，为 EnOS 物联网平台、SCADA 数据采集、企业办公及跨区域协作提供低延迟、高可靠的网络底座 |
| 对应痛点 | **#1 多地区网络架构碎片化**（S0-紧急）— 各区域网络独立建设，缺乏统一规划，跨区域互联延迟高、链路不稳定 |
| 上下游关系 | **上游依赖：** 业务架构团队提供流量模型和延迟要求<br>**下游影响：** 直接影响 #3 灾备能力（DR 需要跨区域网络）、#6 监控可观测性（需要网络可达性） |
| 能力稀缺度 | **极高** — 同时具备全球化网络规划经验和能源行业 OT 网络理解的人才极为稀缺 |
| 岗位权重 | 八大能力域中权重最高（30%），因为网络是所有其他能力的物理基础 |

## 核心职责详解 {hyp}

### 1.1 — 全球网络拓扑设计
根据远景能源全球业务分布，设计 Hub-Spoke 或 Mesh 的全球网络拓扑，确定各区域间的互联方式（专线 / VPN / SD-WAN）和流量路由策略
✅ | 能够画出完整的全球网络拓扑图，标注每条链路的带宽、延迟、冗余方式和成本；能够解释为什么选择当前拓扑而不是其他方案 | → 拓扑设计文档<br>→ 成本对比分析
❌ | 只能描述单个区域的网络，无法从全局视角解释区域间互联策略；对运营商选择和跨国专线缺乏经验 | → 网络拓扑评审
→ | 安排全球网络架构培训；引入外部网络咨询顾问协助设计 |

### 1.2 — 云互联架构（Cloud Interconnect / ExpressRoute / 专线）
管理与阿里云、AWS、GCP 的物理/虚拟互联，包括 Dedicated Interconnect、Partner Interconnect、Direct Connect、Express Route 的选型、容量规划和 SLA 管理
✅ | 熟练配置至少 2 种云厂商的专线互联；能够解释 10G/100G 物理端口、VLAN attachment、BGP session 的配置要点和故障排查流程 | → 互联配置文档<br>→ 故障排查 SOP
❌ | 依赖云厂商 PS 完成互联配置；对 BGP peering、AS path、MED/LocalPref 等路由属性不熟悉 | → 云互联实操培训
→ | 联合云厂商进行互联配置 workshop，逐步建立自主能力 |

### 1.3 — SD-WAN 与混合云网络
评估和部署 SD-WAN 解决方案（如 Cisco Viptela、Fortinet、VMware VeloCloud），实现多分支办公室和工厂的智能组网，并与云 VPC 打通
✅ | 能够对比 3+ 种 SD-WAN 方案的功能矩阵（加密方式、QoS、应用识别、零信任集成）；有实际 SD-WAN 部署和运维经验 | → SD-WAN 选型报告<br>→ 部署方案
❌ | 对 SD-WAN 只有概念理解，未参与过实际部署；不了解 SD-WAN 与传统 MPLS 的混合组网方案 | → SD-WAN PoC 评估
→ | 安排 SD-WAN 厂商 PoC，选择 1 个区域试点 |

### 1.4 — DNS 与全球负载均衡
设计全球 DNS 架构（GeoDNS / Anycast DNS），配合 GSLB（全球服务器负载均衡）实现跨区域流量调度和故障切换
✅ | 能够设计分层 DNS 架构（根 → 区域 → 本地缓存），配置 DNS 策略路由（延迟、地理位置、权重）；理解 DNS 在 DR 切换中的关键作用 | → DNS 架构设计<br>→ DR 切换 SOP
❌ | DNS 知识停留在基础解析层面，不了解 Anycast、EDNS Client Subnet、DNS-based load balancing 等高级特性 | → DNS 架构培训
→ | 引入 DNS 专家或使用托管 DNS 服务（如 Cloudflare/Route53）简化管理 |

### 1.5 — 网络安全与合规
在网络层面实施安全策略：防火墙规则、IDS/IPS、DDoS 防护、网络分段（微分段）、加密传输，确保满足各国网络安全法规
✅ | 能够设计基于零信任原则的网络分段策略；熟悉 NSG/Security Group、WAF、DDoS 防护的配置和运维；了解等保 2.0 对网络安全的要求 | → 网络安全架构<br>→ 合规 checklist
❌ | 网络安全知识停留在基础防火墙规则层面，不了解微分段、零信任网络、东西向流量监控 | → 网络安全培训
→ | 与安全团队紧密协作，网络架构设计阶段即引入安全评审 |

### 1.6 — 工业 OT 网络集成
理解风电场 SCADA 网络、IoT 设备网络（EnOS 边缘网关）的特殊需求，设计 IT/OT 网络融合架构，确保低延迟和高可靠性
✅ | 了解 OT 网络协议（Modbus、OPC UA、MQTT）；能够设计 IT/OT 网络隔离与安全互联方案（DMZ、数据二极管）；理解实时控制对网络延迟的极端要求 | → IT/OT 网络架构<br>→ 安全隔离方案
❌ | 只有 IT 网络背景，对 OT 网络协议和工业控制系统完全不了解 | → OT 网络培训
→ | 联合 OT 团队进行跨领域培训；参考 Purdue 模型设计 IT/OT 融合架构 |

## 硬技能要求 {disc}
| 路由与交换 | BGP（eBGP/iBGP）、OSPF、IS-IS 路由协议深度理解；VLAN/VXLAN 二层技术；ECMP 负载均衡；路由策略（Route-Map、Prefix-List、Community） |
| 云网络 | 至少精通 2 种云厂商网络（VPC/VNet 设计、Subnet 规划、Security Group、NAT Gateway、Transit Gateway/VPN Gateway、Peering/Shared VPC） |
| 专线互联 | Dedicated Interconnect / Direct Connect / ExpressRoute 配置与故障排查；Cross-Connect、LOA、Meet-Me Room 等物理互联流程 |
| SD-WAN | 至少 1 种 SD-WAN 平台实操经验；ZTP 部署、应用级 QoS、加密隧道、WAN 优化 |
| 网络安全 | Firewall（Palo Alto/Fortinet/CheckPoint）、IDS/IPS、DDoS 防护、网络微分段、TLS/IPSec 加密 |
| DNS/CDN | DNS 架构设计（BIND/PowerDNS 或托管服务）；CDN 加速策略；GSLB 配置 |
| 监控与排查 | 网络监控工具（SNMP、NetFlow/sFlow、Wireshark、tcpdump）；延迟和丢包诊断方法论 |
| 自动化 | 网络自动化（Ansible/Nornir、Netconf/YANG、Terraform for networking）；配置版本管理 |

## 软技能要求 {disc}
| 运营商管理 | 与电信运营商（中国电信/移动/联通、NTT、Telia、Equinix）谈判合同、协调工期、管理 SLA |
| 跨团队协作 | 与安全团队、应用团队、OT 团队、云厂商多方协调，推动网络变更的评审和审批 |
| 架构布道 | 向非技术背景的管理层（CIO/CFO）清晰解释网络架构决策的业务价值和风险 |
| 成本意识 | 能够在网络性能和成本之间做出合理取舍；具备专线/SD-WAN/VPN 的 TCO 分析能力 |
| 合规理解 | 了解各国对网络基础设施的监管要求（如中国的网络安全法对跨境数据传输的限制） |

## 评估维度 — 面试考察要点 {hyp}

### E1 — 全球网络设计能力
请描述一个你设计的跨 3 个以上国家/区域的网络架构项目
✅ | 能够详细描述拓扑选择（Hub-Spoke vs Full Mesh）、运营商选择、带宽规划、冗余设计、成本优化的完整决策过程；有量化的延迟/吞吐改善数据 | → 架构设计经验
❌ | 只能描述单区域网络，或者依赖系统集成商完成设计；无法解释设计取舍 | → 经验不足
→ | 如候选人在此维度弱，需要评估其学习速度和是否有 mentor 支持 |

### E2 — 故障排查深度
请描述一次跨区域网络故障的排查过程
✅ | 能够描述从现象到根因的完整排查链：用了什么工具（traceroute/mtr/iperf/tcpdump/NetFlow）、如何分层定位（应用层→传输层→网络层→物理层）、如何与运营商协同排查 | → 排查能力
❌ | 排查方法单一（只会 ping 和 traceroute），不会分析数据包、不了解 BGP 路由排查 | → 排查能力弱
→ | 安排 2 周网络排查实战训练营 |

### E3 — 能源行业 IT/OT 理解
你如何看待 IT 网络和 OT（工业控制）网络的融合？
✅ | 了解 Purdue 模型、ISA/IEC 62443 标准；能够解释 IT/OT 网络隔离的必要性（安全 + 实时性）和融合的趋势（边缘计算、MQTT）；有 OT 网络相关项目经验 | → 行业理解
❌ | 对 OT 网络完全不了解，无法区分 IT 网络和 OT 网络的需求差异 | → 需行业培训
→ | 此项可通过入职后的 OT 团队交叉培训弥补，非淘汰项 |

### E4 — 云原生网络
请解释 VPC Peering、Transit Gateway 和 Shared VPC 的区别和适用场景
✅ | 能够清晰对比三者的路由传播方式、带宽限制、成本差异、管理复杂度，并结合远景多云场景推荐最优方案；了解 Service Mesh 和 Zero Trust Network Access | → 云网络能力
❌ | 只知道基本 VPC 创建，不了解多 VPC 互联策略和跨云网络方案 | → 云网络培训
→ | 安排云厂商网络深度培训（AWS Advanced Networking / GCP Network Engineer） |

## 远景能源适配要求 {exec}
| 中国区特殊要求 | 1. 了解中国跨境数据传输的网络层限制（GFW、ICP 备案）<br>2. 熟悉中国三大运营商的专线产品和 SLA<br>3. 了解阿里云网络产品（CEN、SAG、Express Connect）<br>4. 等保 2.0 对网络安全等级保护的具体要求 |
| EnOS 平台网络需求 | 1. EnOS 边缘网关 → 云端的数据上传链路设计（MQTT over TLS）<br>2. 全球数十万 IoT 设备的大规模连接管理<br>3. 实时控制指令的低延迟下发通道（< 100ms）<br>4. EnOS 多租户场景下的网络隔离 |
| 风电场网络特殊性 | 1. 风电场通常位于偏远地区，网络接入条件差（可能只有 4G/5G 或卫星链路）<br>2. SCADA 系统对网络中断极度敏感（可能导致风机停机）<br>3. 海上风电场的网络连接更加特殊（海底光缆 + 微波链路冗余） |
| 多云互联需求 | 1. 中国区（阿里云）↔ 欧洲区（AWS/GCP）的跨云互联<br>2. 各区域私有云 ↔ 公有云的混合连接<br>3. 跨云 DNS 解析和服务发现 |

## 典型工作场景 {exec}
| 场景一：全球骨干网络重构 | **背景：** 远景在 8 个国家的网络各自独立，每个区域自行采购 VPN/专线，没有统一规划<br>**任务：** 设计统一的全球网络架构，选择 Hub 节点（如上海和法兰克福），部署骨干网络互联<br>**挑战：** 各区域已有的网络供应商合同处理、迁移期间的业务不中断、成本预算控制<br>**预期产出：** 全球网络拓扑设计文档、运营商选型报告、分阶段迁移计划、TCO 分析 |
| 场景二：新区域网络接入 | **背景：** 远景在印度新建制造基地，需要将其接入全球网络<br>**任务：** 2 个月内完成印度区域的网络规划、运营商选择、专线部署、与全球骨干网打通<br>**挑战：** 印度运营商 SLA 不可靠、跨国专线延迟较高、需满足印度本地数据法规<br>**预期产出：** 印度区域网络架构设计、运营商合同签署、端到端连通性测试报告 |
| 场景三：海上风电场网络覆盖 | **背景：** 远景在北海部署 50+ 台海上风机，需要可靠的网络连接支持 SCADA 监控<br>**任务：** 设计海上风电场 → 陆地基站 → 云端的全链路网络方案<br>**挑战：** 海上环境恶劣、光缆铺设成本高、需要微波链路作为备份、极端天气下的网络可靠性<br>**预期产出：** 海上网络架构设计、冗余方案、应急通信预案 |

## 建议学习路径 {exec}
| Phase 1（入职 1-3 月） | 1. 全面了解远景现有网络架构（各区域拓扑、运营商合同、带宽利用率）<br>2. 与 OT 团队交叉培训，理解风电场 SCADA 网络需求<br>3. 完成 AWS Advanced Networking 或 GCP Network Engineer 认证<br>4. 梳理各区域网络痛点清单 |
| Phase 2（4-6 月） | 1. 完成全球网络架构设计<br>2. 启动骨干网络改造项目（关键路径）<br>3. 建立网络性能基线和监控体系<br>4. 与运营商建立长期合作关系 |
| Phase 3（7-12 月） | 1. 完成骨干网络改造和区域互联部署<br>2. 实现网络自动化运维（配置管理、变更审计）<br>3. 建立网络容量规划和成本优化机制<br>4. 沉淀网络架构设计模板和最佳实践文档 |

## 成功标准与 KPI {risk}
| 成功标准 | 1. 跨区域网络延迟（P99）< 200ms（当前 > 2000ms）<br>2. 骨干网络可用性 > 99.95%<br>3. 全球网络拓扑文档化率 100%<br>4. 网络变更流程 100% 通过自动化审批<br>5. 网络成本年度节省 15%+（通过运营商整合和流量优化） |
| 风险 | 1. 骨干网络改造是全局关键路径，延期将连锁影响 #3 灾备<br>2. 中国跨境网络受政策影响大，方案可能需随政策调整<br>3. 海上风电场网络环境特殊，标准方案可能不适用 |
| 与其他能力域的协同 | **→ 能力三（安全合规）：** 网络架构设计阶段即需引入安全评审，确保网络分段和加密策略满足合规要求<br>**→ 能力六（容灾）：** 网络冗余设计直接决定 DR 切换的 RTO<br>**→ 能力七（可观测性）：** 网络监控是整体可观测性平台的基础层 |
