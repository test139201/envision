## 基本信息
| 能力域定位 | 建立远景能源跨阿里云/AWS/GCP 的统一成本管理体系，实现成本可视化、归因分析、优化执行和持续治理的 FinOps 闭环 |
| 对应痛点 | **#5 云成本失控**（S1-重要）— 多云环境缺乏统一 FinOps 治理，资源利用率低，云支出增速远超业务增速 |
| 上下游关系 | **上游依赖：** 能力二（多云管理）提供多云资源元数据和标签体系<br>**下游影响：** 成本数据为管理层决策提供量化依据，直接影响 IT 预算分配 |
| 能力稀缺度 | **中** — FinOps 专业人才逐渐增多，但具备多云（尤其阿里云）FinOps 经验的人才较少 |
| 岗位权重 | 八大能力域中权重 8%，但对管理层可见度最高（直接影响财务指标） |

## 核心职责详解 {hyp}

### 5.1 — 多云成本可视化
建立跨云统一成本看板，实现按业务线/项目/环境/区域/资源类型多维度的成本分解与展示
✅ | 有跨云成本归集经验；能够将阿里云/AWS/GCP 的账单数据统一到一个平台；实现实时和历史成本趋势分析；支持按自定义标签进行成本分配 | → 成本看板<br>→ 数据管道
❌ | 各云账单独立查看，没有统一视图；成本数据滞后严重（月结后才能看到） | → 可视化建设
→ | 从 CUR（AWS）/ 账单 API（阿里云）/ Billing Export（GCP）开始，汇聚到统一数据仓库 |

### 5.2 — 成本归因与 Chargeback/Showback
建立成本归因体系：通过标签策略将每一笔云支出归因到具体的业务线、项目、团队，实现 Chargeback（实际收费）或 Showback（展示透明）
✅ | 有标签策略设计和推行经验；能够设计多级标签体系（business-unit/project/environment/cost-center）；有 Chargeback 模型设计和执行经验 | → 标签策略<br>→ Chargeback 模型
❌ | 云资源没有或很少有标签，成本无法归因；各团队对自己的云支出没有感知 | → 标签治理
→ | 先定义必需标签集（mandatory tags），再通过 Policy 强制执行 |

### 5.3 — 资源优化（Right-sizing / 预留 / Spot）
持续识别和执行资源优化机会：闲置资源清理、过度配置的降配（Right-sizing）、预留实例/承诺使用折扣（RI/CUD/SP）、Spot/抢占式实例利用
✅ | 有系统化的资源优化方法论（识别 → 分析 → 决策 → 执行 → 验证）；精通各云的折扣机制（RI/SP/CUD/按量/Spot）；有 20%+ 成本优化的实际案例 | → 优化方法<br>→ 节省案例
❌ | 优化只停留在"关掉不用的机器"，不了解预留购买策略和自动化优化 | → 优化培训
→ | 引入 FinOps 工具（如 Spot.io/CloudHealth/阿里云 Advisor）辅助优化 |

### 5.4 — 预算管理与异常告警
建立云支出预算管理体系：年度预算编制、月度预算跟踪、成本异常实时告警
✅ | 有云预算管理经验；能够基于历史数据和业务增长预测编制年度预算；配置了多级告警（80%/90%/100% 预算水位线） | → 预算模型<br>→ 告警规则
❌ | 没有明确的云预算，月底账单出来才知道花了多少钱 | → 预算建设
→ | 利用各云的 Budget 功能（AWS Budgets / 阿里云费用预警 / GCP Budget）快速建立基础告警 |

### 5.5 — FinOps 文化与激励机制
推动 FinOps 文化在全公司落地：让工程团队具备成本意识，建立成本优化的激励和考核机制
✅ | 有推动 FinOps 文化变革的经验；建立过成本优化激励机制（如节省金额的一定比例回馈团队）；定期举办成本优化分享会 | → FinOps 文化<br>→ 激励机制
❌ | 成本管理只是 IT 财务的事，工程团队不关心也不了解成本 | → 文化建设
→ | 从成本 Showback 开始培养意识，让每个团队看到自己的花费 |

## 硬技能要求 {disc}
| 成本分析 | 精通各云账单结构（AWS CUR、阿里云账单 API、GCP Billing Export）；成本分解和归因方法论；趋势分析和异常检测 |
| 折扣机制 | AWS（RI/SP/Savings Plans）、阿里云（包年包月/预留实例/节省计划/抢占式实例）、GCP（CUD/Spot VMs）的购买策略和 ROI 计算 |
| FinOps 工具 | CloudHealth / Spot.io / Kubecost / 阿里云成本管家 / AWS Cost Explorer / GCP Cost Management |
| 标签治理 | 多云标签策略设计、标签合规检查自动化（AWS Tag Policies / 阿里云标签策略 / GCP Label）、未标签资源识别 |
| 数据分析 | SQL 查询成本数据、BI 工具可视化（Grafana/Superset/QuickSight）、成本预测模型 |
| K8s 成本 | Kubernetes 多租户成本分摊（Kubecost/OpenCost）、Namespace/Label 级别成本归因 |

## 评估维度 — 面试考察要点 {hyp}

### E1 — 成本优化成果
请描述你实现的最大一次成本优化（金额和百分比）
✅ | 能够描述完整的优化过程：发现机会 → 分析 ROI → 获取审批 → 执行变更 → 验证效果；有 20%+ 的成本节省案例；不仅包括"关机"这种低级优化 | → 优化经验
❌ | 没有量化的成本优化案例，或者优化只是偶尔的临时动作 | → 需培养
→ | FinOps 实操能力可快速培养，关键是分析思维和数据敏感度 |

### E2 — 预留购买决策
你如何制定预留实例/承诺使用折扣的购买策略？
✅ | 能够解释覆盖率（Coverage）vs 利用率（Utilization）的分析方法；了解全预付/部分预付/无预付的选择策略；有基于使用趋势的购买决策框架 | → 折扣策略
❌ | 不了解预留购买的分析方法，或者全靠猜测购买 | → 需培训
→ | AWS Compute Optimizer 和阿里云 Advisor 可提供推荐，但决策能力需要培养 |

### E3 — 跨组织推动
FinOps 需要全公司参与，你如何让不关心成本的工程团队配合？
✅ | 有组织变革推动经验；用过 Showback 报告让团队看到自己的花费；建立过正向激励机制；知道如何避免"成本警察"的负面印象 | → 推动能力
❌ | 只会强制要求或发邮件催促，缺乏策略性的推动方法 | → 需软技能
→ | 推荐 FinOps Foundation 的培训和社区，学习行业最佳实践 |

## 远景能源适配要求 {exec}
| 多云成本复杂性 | 远景的云成本涉及：<br>1. 阿里云（中国区最大支出）— 包年包月为主，计费模式与 AWS 差异大<br>2. AWS（海外最大支出）— On-Demand + RI + SP 混合<br>3. GCP（数据分析）— BigQuery 按查询量计费，成本波动大<br>4. 私有云（工厂）— 折旧+运维成本，需统一到 TCO 计算 |
| EnOS 平台成本 | EnOS 是远景最大的云资源消费者：<br>1. 全球数十万 IoT 设备的数据接入和存储<br>2. 实时数据处理的计算资源<br>3. 多区域部署的冗余成本<br>4. 需要建立 per-device 或 per-tenant 的成本模型 |
| IoT 数据存储成本 | 风机/光伏/电池遥测数据持续增长：<br>1. 热数据（实时查询）→ 高性能存储<br>2. 温数据（7 天内回溯）→ 标准存储<br>3. 冷数据（合规存档）→ 归档存储<br>4. 数据分层策略可节省 40-60% 存储成本 |

## 典型工作场景 {exec}
| 场景一：年度云预算编制 | **背景：** CIO 要求提交下一财年的云支出预算<br>**任务：** 基于历史数据、业务增长预测、优化计划，编制分区域/分云的年度预算<br>**挑战：** IoT 设备增长带来的存储成本难以精确预测；汇率波动影响海外区域预算<br>**预期产出：** 分月/分区域/分云的预算表 + 关键假设说明 + 优化空间分析 |
| 场景二：紧急成本异常排查 | **背景：** 本月阿里云账单较上月增长 40%，超出预算水位线<br>**任务：** 快速定位成本增长的根因（新增资源？价格变更？异常流量？）<br>**挑战：** 阿里云账单细粒度数据获取不如 AWS 方便；需要跨团队确认业务变更<br>**预期产出：** 根因分析报告 + 优化建议 + 预算调整方案 |
| 场景三：RI/SP 购买策略制定 | **背景：** AWS 海外区域有大量 On-Demand EC2，需要通过 RI/SP 降低成本<br>**任务：** 分析过去 6 个月的使用趋势，制定购买策略<br>**挑战：** 部分实例使用不稳定（开发/测试环境）；需要区分适合 RI 和 SP 的工作负载<br>**预期产出：** 购买清单 + 预期节省金额 + 风险说明 |

## 建议学习路径 {exec}
| Phase 1（入职 1-3 月） | 1. 导出过去 6-12 个月全部云账单，建立成本基线<br>2. 获取 FinOps Foundation Certified Practitioner 认证<br>3. 盘点现有标签覆盖率和质量<br>4. 识别 Top 10 快速优化机会 |
| Phase 2（4-6 月） | 1. 上线多云成本看板<br>2. 推行标签策略（mandatory tags + 合规检查）<br>3. 执行首批优化（闲置清理 + Right-sizing + RI 购买）<br>4. 建立成本异常告警体系 |
| Phase 3（7-12 月） | 1. 实现 Chargeback/Showback 到业务线<br>2. 数据存储分层优化（热/温/冷）<br>3. K8s 成本精细化（per-namespace/per-pod）<br>4. 建立 FinOps 月报机制和优化激励 |

## 成功标准与 KPI {risk}
| 成功标准 | 1. 云成本同比节省 25%+<br>2. 资源利用率从 < 20% 提升到 > 50%<br>3. 标签覆盖率 > 95%（生产资源）<br>4. 成本异常发现时间 < 24h（实时告警）<br>5. 每个业务线可看到自己的月度成本报告 |
| 风险 | 1. 过度优化可能影响系统性能和可靠性（如过度 Right-sizing）<br>2. 标签治理需要全公司配合，推动阻力大<br>3. 阿里云的成本分析能力不如 AWS 成熟，数据获取可能受限 |
| 与其他能力域的协同 | **→ 能力二（多云）：** CMP 平台提供多云资源元数据和标签数据<br>**→ 能力四（IaC）：** IaC 模板中内置成本标签<br>**→ 能力八（团队）：** FinOps 培训纳入团队技能发展计划 |
