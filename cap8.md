## 基本信息
| 能力域定位 | 建设和管理一支全球分布式的基础设施运维团队，弥合各区域间的能力差距，建立持续学习和协作的文化 |
| 对应痛点 | **#9 运维团队技能差距**（S2-中等）— 团队在云原生技术栈方面能力薄弱，各区域能力参差不齐 |
| 上下游关系 | **上游依赖：** 所有其他 7 个能力域都需要团队来执行<br>**下游影响：** 团队能力是所有技术能力落地的前提条件 |
| 能力稀缺度 | **中高** — 技术管理能力 + 跨文化管理经验 + 基础设施深度技术背景的组合较少 |
| 岗位权重 | 八大能力域中权重 9%，但对长期可持续性影响最大 |

## 核心职责详解 {hyp}

### 8.1 — 全球团队组织设计
设计全球基础设施团队的组织架构：中央集权 vs 联邦制 vs 混合模式；确定各区域的人员配置和职责分工
✅ | 能够设计适合远景的组织模型：中央 CoE（Center of Excellence）+ 区域执行团队；定义中央和区域的职责边界；有跨国团队管理经验 | → 组织设计<br>→ 职责矩阵
❌ | 缺乏组织设计经验，只能管理单一地点的小团队；不了解分布式团队管理的挑战 | → 管理培训
→ | 学习成熟公司的全球 IT 组织模式（Google SRE / Netflix 平台团队模型） |

### 8.2 — 跨时区 On-Call 体系
设计 Follow-the-Sun 的 On-Call 轮值体系，确保 7×24 全球覆盖，同时不过度消耗团队精力
✅ | 有 Follow-the-Sun On-Call 设计经验；能够设计合理的轮值频率（每人每月 On-Call 天数上限）、升级策略（L1→L2→Manager）、补偿机制 | → On-Call 设计<br>→ 升级流程
❌ | On-Call 靠"谁在线谁就响应"的随机模式，没有正式轮值；或者少数人承担大部分 On-Call | → 轮值建设
→ | 先建立正式的 On-Call 轮值表（PagerDuty/OpsGenie），再优化覆盖策略 |

### 8.3 — 技能发展与认证体系
建立团队技能发展路径：技能评估 → 差距识别 → 培训计划 → 实战演练 → 认证考核
✅ | 有技能矩阵设计经验（按能力域 × 熟练度评估每个人）；能够制定个性化发展计划（IDP）；推动团队获取云厂商认证（AWS SAA/SAP、阿里云 ACP、GCP ACE/PCA） | → 技能矩阵<br>→ 培训计划
❌ | 没有系统化的技能评估和培训体系，团队成长靠自学 | → 培训体系
→ | 先做技能评估（问卷 + 实操测试），识别最急需的技能差距 |

### 8.4 — 知识管理与文档体系
建立团队的知识管理体系：Runbook、SOP、架构文档、故障复盘、最佳实践的标准化管理
✅ | 有知识管理平台建设经验（Confluence/Notion/Docusaurus）；能够建立文档标准（模板、审核流程、更新机制）；文档覆盖率高且保持更新 | → 知识库<br>→ 文档标准
❌ | 知识散落在个人脑中或聊天记录中，关键人离职会导致知识断层 | → 知识建设
→ | 从 Runbook（运维手册）开始，确保关键操作都有标准文档 |

### 8.5 — 招聘与人才吸引
在各区域招聘合适的基础设施工程师，建立远景作为技术雇主品牌的吸引力
✅ | 有技术招聘全流程经验（JD 编写、筛选、技术面试、Offer 谈判）；了解各区域的人才市场和薪酬水平；有雇主品牌建设意识（技术博客、开源贡献、技术分享） | → 招聘流程<br>→ 雇主品牌
❌ | 招聘完全依赖 HR，技术面试缺乏标准化；不了解如何吸引优秀技术人才 | → 招聘优化
→ | 建立标准化技术面试题库和评分标准 |

### 8.6 — 跨文化团队管理
管理跨中国、欧洲、东南亚、日本的多文化团队，处理文化差异带来的沟通和协作挑战
✅ | 有跨文化团队管理经验；理解不同文化的工作风格差异（如中国团队的执行力 vs 北欧团队的自主性 vs 日本团队的流程导向）；能够建立包容性的团队文化 | → 文化理解<br>→ 管理风格
❌ | 只有单一文化背景的管理经验，不了解跨文化管理的挑战 | → 文化学习
→ | 入职后到各区域出差，面对面了解各团队的文化和工作方式 |

## 硬技能要求 {disc}
| 组织设计 | 全球 IT 组织模型（中央 CoE + 联邦 + 混合）；RACI 矩阵；职责定义和人员配置规划 |
| On-Call 管理 | PagerDuty / OpsGenie 配置；On-Call 轮值设计；升级策略；SLA 管理；On-Call 补偿政策 |

| 知识管理 | 知识库平台选型和运营；文档模板标准化；Runbook/SOP 编写规范；知识审核和更新机制 |
| 绩效管理 | OKR/KPI 体系设计；技术绩效评估标准；1:1 沟通方法论；人才盘点（9-box grid） |

## 软技能要求 {disc}
| 跨文化领导力 | 理解和尊重文化差异；建立包容性团队氛围；调适管理风格适应不同文化背景的团队成员 |
| 沟通能力 | 英文工作语言流利（全球团队通用语言）；中文母语（中国团队沟通）；能够用简洁清晰的方式传达技术和管理决策 |
| 人才识别 | 识别高潜力人才（HIPO）并给予发展机会；识别绩效不足的原因（能力、意愿、环境）并针对性解决 |
| 变革管理 | 推动团队从传统运维向云原生 SRE 转型的变革管理能力；处理团队对变革的抵触情绪 |
| 影响力 | 向上管理（CIO）：争取预算和资源；横向影响（其他部门）：推动跨部门协作 |

## 评估维度 — 面试考察要点 {hyp}

### E1 — 全球团队管理经验
请描述你管理过的最大规模的分布式技术团队
✅ | 管理过 3+ 国家/时区的分布式团队（20+ 人）；能具体描述如何解决跨时区协作、文化差异、沟通效率的挑战；有可量化的团队成果 | → 管理经验
❌ | 只管理过同一地点的小团队（< 10 人），没有跨国管理经验 | → 经验不足
→ | 跨国管理经验可以通过先管理多地点（如北京+上海）来积累 |

### E2 — 团队转型推动
你如何推动一个传统运维团队向云原生 SRE 转型？
✅ | 有实际推动团队转型的经验；能够描述渐进式策略：1) 认知转变（培训+分享）2) 工具转变（自动化+IaC）3) 流程转变（On-Call+Postmortem）4) 文化转变（blameless+持续改进）；处理过团队抵触并成功转化 | → 转型经验
❌ | 没有推动过团队变革，或者变革失败未做复盘 | → 需培养
→ | 学习变革管理方法论（Kotter 8-Step / ADKAR）并在小范围试点 |

### E3 — 人才培养
你如何培养团队成员的技术能力？有具体案例吗？
✅ | 有系统化的人才培养方法：技能评估 → IDP → 培训+实战 → 认证 → 复盘；能够举出将初级工程师培养为独当一面的高级工程师的具体案例 | → 培养能力
❌ | 人才培养靠"自己看文档"或"跟着做就行"，缺乏系统化方法 | → 需方法论
→ | 建立 Mentor 制度，每个新人配一个技术 Mentor |

## 远景能源适配要求 {exec}
| 团队现状 | 远景全球基础设施团队现状：<br>1. 中国区团队（~15 人）：云能力相对较强，但偏阿里云<br>2. 欧洲区团队（~5 人）：传统 IT 背景为主，云经验少<br>3. 东南亚/日本团队（~3 人）：规模小，依赖中国区支持<br>4. 总计 ~25 人，需扩编至 ~35 人 |
| 能力差距 | 最急需补齐的能力：<br>1. Kubernetes/容器编排（全球普遍薄弱）<br>2. Terraform/IaC（欧洲/东南亚几乎为零）<br>3. 可观测性平台运维（只有少数人会 Prometheus/Grafana）<br>4. 安全合规（缺乏跨国合规经验）<br>5. 英文沟通（中国团队英文能力参差不齐） |
| 文化挑战 | 1. 中国团队习惯加班文化 vs 欧洲团队严格遵守工时<br>2. 沟通风格差异：中国团队偏含蓄、日本团队偏正式、北欧团队偏直接<br>3. 决策方式差异：中国团队倾向自上而下、北欧团队期望参与式决策<br>4. 需要找到尊重差异同时确保效率的平衡点 |
| 扩编需求 | 需要新增的关键岗位：<br>1. 全球网络架构师 × 1<br>2. SRE / 可观测性工程师 × 2<br>3. 安全合规工程师 × 2（中国 + 欧洲各 1）<br>4. FinOps 分析师 × 1<br>5. 各区域 IaC 工程师 × 3 |

## 典型工作场景 {exec}
| 场景一：技能评估与培训计划制定 | **背景：** 入职后需要全面了解全球团队的能力现状<br>**任务：** 设计技能评估方案，执行评估，制定分层培训计划<br>**挑战：** 各区域技能基准不同；培训资源和时间有限；需要平衡培训与日常运维<br>**预期产出：** 全球技能矩阵、Top 5 急需技能、12 月培训计划 |
| 场景二：跨区域协作流程建立 | **背景：** 当前各区域独立运维，缺乏标准化流程和协作机制<br>**任务：** 建立统一的运维流程（变更管理、事件管理、问题管理）和协作工具链<br>**挑战：** 各区域已有自己的习惯和工具；需要在统一和灵活之间平衡<br>**预期产出：** 统一运维流程文档、协作工具链上线、首个跨区域 On-Call 轮值周期运行 |
| 场景三：关键岗位招聘 | **背景：** 需要在 3 个月内招聘全球网络架构师和欧洲区安全合规工程师<br>**任务：** 编写 JD、定义面试流程、筛选候选人、技术面试、发放 Offer<br>**挑战：** 全球网络架构师市场稀缺；欧洲区安全合规需了解当地法规；远景在技术人才市场的品牌认知度不高<br>**预期产出：** 2 个关键岗位在 3 个月内到岗 |

## 建议学习路径 {exec}
| Phase 1（入职 1-3 月） | 1. 到各区域出差，面对面了解团队成员和工作环境<br>2. 完成全球团队技能评估<br>3. 建立与各区域 lead 的 1:1 沟通机制<br>4. 确定组织架构调整方案 |
| Phase 2（4-6 月） | 1. 启动关键岗位招聘<br>2. 建立统一运维流程和工具链<br>3. 启动首批培训（K8s + Terraform 实战训练营）<br>4. 建立跨时区 On-Call 轮值 |
| Phase 3（7-12 月） | 1. 完成团队扩编<br>2. > 60% 核心成员获得云认证<br>3. 知识库和 Runbook 覆盖核心场景<br>4. 团队满意度和保留率达标 |

## 成功标准与 KPI {risk}
| 成功标准 | 1. 团队云认证率 > 60%（核心成员）<br>2. On-Call MTTR < 15 分钟（全球平均）<br>3. 关键岗位空缺率 < 10%<br>4. 团队 eNPS（员工推荐度）> 30<br>5. 知识库文档覆盖核心场景 > 80%<br>6. 跨区域协作流程 100% 标准化 |
| 风险 | 1. 关键人才难以招聘或流失<br>2. 培训投入短期内不见效果，管理层可能质疑 ROI<br>3. 文化差异导致沟通效率低<br>4. 运维团队抵触从传统向云原生的转型 |
| 与其他能力域的协同 | **→ 所有能力域：** 团队是所有技术能力落地的执行主体<br>**→ 能力四（IaC）：** IaC 培训是团队技能提升的核心内容<br>**→ 能力五（FinOps）：** FinOps 培训纳入团队能力建设<br>**→ 能力七（可观测性）：** SRE 实践培训是团队转型的关键 |
