## 基本信息
| 能力域定位 | 设计和运营跨阿里云（中国）、AWS（全球）、GCP（全球）和私有云（部分工厂）的多云管理体系，实现资源统一编排、一致性策略、跨云服务互通 |
| 对应痛点 | **#10 供应商锁定风险**（S3-关注）直接相关；同时影响 #4 IaC、#5 成本、#6 监控等多个痛点的跨云一致性 |
| 上下游关系 | **上游依赖：** 业务部门提供工作负载分布需求和数据主权要求<br>**下游影响：** 为 #4 IaC 提供多云编排基础，为 #5 FinOps 提供多云成本归集能力 |
| 能力稀缺度 | **高** — 熟悉阿里云 + AWS/GCP 的复合型人才在市场上较少；多数候选人只精通 1-2 种云 |
| 岗位权重 | 八大能力域中权重第二（20%），因为远景能源的多云现实决定了管理复杂度 |

## 核心职责详解 {hyp}

### 2.1 — 多云架构策略制定
制定远景能源的多云策略：哪些工作负载放在哪朵云、为什么这样放、如何避免过度锁定又不过度分散
✅ | 能够制定清晰的工作负载分布矩阵（考虑数据主权、延迟、成本、供应商风险）；能解释"right cloud for right workload"的决策框架 | → 多云策略文档<br>→ 工作负载矩阵
❌ | 对多云策略只有模糊的"避免锁定"概念，无法给出具体的工作负载分布决策依据 | → 策略培训
→ | 组织多云策略研讨，邀请各云厂商 SA 参与，共同制定决策框架 |

### 2.2 — 云管理平台（CMP）建设
选型或自建统一的云管理平台，实现多云资源的可视化、编排、成本、安全的集中管理
✅ | 有 CMP 平台建设经验（如 Morpheus、CloudBolt、ServiceNow ITOM、自建）；能够对比 SaaS CMP vs 自建的 TCO 和适用场景；平台覆盖率 > 80% 云资源 | → CMP 选型报告<br>→ 平台架构
❌ | 每朵云用各自的控制台管理，没有统一视图；对 CMP 市场不了解 | → CMP 评估
→ | 启动 CMP 选型 PoC，优先选择支持阿里云 + AWS/GCP 的平台 |

### 2.3 — 跨云网络互通与服务编排
确保多云环境之间的网络互通（Transit VPC、Cloud Interconnect、VPN Mesh），以及跨云的服务发现和 API 网关
✅ | 能够设计跨云 Transit 架构，打通阿里云 VPC ↔ AWS VPC ↔ GCP VPC；了解跨云 DNS 解析、Service Mesh（Istio multi-cluster）、API Gateway 联邦 | → 跨云互联方案<br>→ 服务编排架构
❌ | 各云网络完全隔离，靠 Public Internet 通信；不了解 Transit 架构和跨云服务发现 | → 跨云网络培训
→ | 从最关键的两朵云（阿里云 ↔ AWS）开始，建立基础互联通道 |

### 2.4 — 多云身份联邦与权限治理
建立统一的身份管理体系，实现 SSO 跨云登录、统一 RBAC 策略、跨云审计日志汇聚
✅ | 有 SAML/OIDC 联邦配置经验；能够设计统一的 RBAC 模型映射到各云厂商的 IAM 体系（阿里云 RAM、AWS IAM、GCP IAM）；实现审计日志集中化 | → 身份联邦方案<br>→ RBAC 模型
❌ | 各云独立的账号体系，存在大量共享账号，权限分散且不一致 | → IAM 治理
→ | 先统一到企业 IdP（如 Azure AD/Okta），再逐步推动 RBAC 标准化 |

### 2.5 — 多云数据管理与迁移
管理跨云的数据存储策略（热/温/冷分层）、数据复制与同步、跨云数据迁移方案
✅ | 了解各云的存储产品差异（S3/OSS/GCS、EBS/ESSD/PD）；能够设计跨云数据复制方案（实时同步 vs 异步复制）；有大规模数据迁移（TB/PB 级）经验 | → 数据管理策略<br>→ 迁移方案
❌ | 数据完全绑定在单一云厂商的存储服务上，没有跨云数据管理策略 | → 数据战略培训
→ | 盘点关键数据资产的存储位置和跨云复制需求，制定数据分布策略 |

### 2.6 — 供应商关系管理与合同优化
管理与 3+ 云厂商的商务关系，包括合同谈判、折扣获取、SLA 监控、年度评审
✅ | 有与多家云厂商同时谈判的经验；了解 EDP（Enterprise Discount Program）/ CUD（Committed Use Discount）/ RI（Reserved Instance）等折扣机制；能够利用竞争关系获取更优条件 | → 供应商管理
❌ | 被动接受云厂商报价，不了解折扣谈判策略和竞争杠杆 | → 商务培训
→ | 与采购部门协作，建立云厂商年度评审机制 |

## 硬技能要求 {disc}
| 阿里云 | 精通阿里云核心产品：ECS、OSS、RDS、SLB、VPC、CEN（云企业网）、SAG（智能接入网关）、RAM、ActionTrail、云监控；了解阿里云特有服务（MaxCompute、DRDS、PAI） |
| AWS | 精通 AWS 核心产品：EC2、S3、RDS、ELB、VPC、Transit Gateway、Direct Connect、IAM、CloudTrail、CloudWatch；了解 AWS Organizations 多账号管理 |
| GCP | 熟悉 GCP 核心产品：GCE、GCS、Cloud SQL、Cloud Load Balancing、VPC、Shared VPC、Cloud Interconnect、IAM、Cloud Audit Logs、Cloud Monitoring |
| 容器编排 | Kubernetes 多集群管理（Rancher/Anthos/ACK）；跨云 K8s 联邦；Service Mesh（Istio/Linkerd）跨云配置 |
| IaC 多云 | Terraform 多 Provider 编排（alicloud + aws + google）；Pulumi 或 Crossplane 跨云抽象；Module Registry 管理 |
| 数据管理 | 跨云数据复制工具（Rclone、Datasync、镜像同步）；数据库跨云迁移（DMS、DTS、Database Migration Service） |

## 软技能要求 {disc}
| 供应商谈判 | 与阿里云/AWS/GCP 三方同时维护关系，利用竞争态势争取最优条件；管理 TAM/SA 资源 |
| 架构决策 | 在性能、成本、锁定风险之间做平衡决策；能够用数据和案例说服管理层 |
| 变更管理 | 推动跨云标准化过程中的组织变革管理（各区域团队的习惯和偏好不同） |

## 评估维度 — 面试考察要点 {hyp}

### E1 — 多云实战经验
请描述你管理的最复杂的多云环境（涉及几朵云、多少工作负载、如何协调）
✅ | 管理过 2+ 公有云 + 私有云/on-prem 的混合环境；能具体描述工作负载分布决策、跨云互联方案、统一管理工具链；有可量化的成果（成本节省、故障率下降） | → 多云经验
❌ | 只管理过单一云环境，或者多云只是概念上的分布，实际没有统一管理 | → 经验不足
→ | 评估候选人对多云复杂性的理解深度，单云高手+学习意愿也可考虑 |

### E2 — 阿里云深度
远景中国区重度使用阿里云，请描述你在阿里云上的项目经验
✅ | 有阿里云大规模使用经验（100+ 台 ECS 以上）；了解阿里云特有的产品和限制（如国际站 vs 国内站的差异、RAM 与 AWS IAM 的区别）；有阿里云认证（ACE/ACP） | → 阿里云能力
❌ | 没有阿里云经验，只熟悉 AWS/GCP；不了解中国云市场的特殊性 | → 阿里云学习
→ | 阿里云的学习曲线较低（对有其他云经验的人），可通过 1-2 月密集学习弥补 |

### E3 — 抽象层设计能力
你如何设计一个跨云的应用部署抽象层，让应用团队不需要关心底层是哪朵云？
✅ | 能够描述 Kubernetes + Terraform + GitOps 的跨云抽象方案；了解 Crossplane 等云原生跨云工具；理解抽象粒度的取舍（过度抽象 → 失去各云优势 vs 不够抽象 → 厂商锁定） | → 架构设计能力
❌ | 没有抽象层设计经验，每朵云完全独立的技术栈 | → 架构培训
→ | 从 Terraform Module 标准化开始，逐步建立抽象能力 |

## 远景能源适配要求 {exec}
| 多云现实 | 中国区：阿里云为主（等保合规 + 本地支持），部分使用华为云<br>欧洲区：AWS 为主（成熟生态），部分 GCP（BigQuery 数据分析）<br>东南亚/日本：AWS 为主<br>北美：AWS + GCP 混合<br>工厂/风电场：私有云 + 边缘计算 |
| EnOS 平台部署 | EnOS 需要在多个区域/多朵云上部署独立实例，确保数据主权合规的同时实现全局管理视图 |
| 数据主权约束 | 中国数据不出境（数据安全法）；欧洲数据留在欧洲（GDPR）；但全球管理层需要统一的数据分析视图 → 需要设计数据联邦而非数据集中方案 |
| 边缘计算集成 | 风电场/工厂的边缘节点需要纳入统一管理体系，但网络条件受限 → 需要离线优先的管理策略 |

## 典型工作场景 {exec}
| 场景一：多云统一管理平台建设 | **背景：** 当前 3 朵公有云 + 2 套私有云，共计 5 个独立管理界面，运维效率低<br>**任务：** 选型并部署统一 CMP，实现资源可视化、成本归集、策略一致性<br>**挑战：** 阿里云 API 与 AWS/GCP API 差异大；私有云（VMware/OpenStack）需要额外适配<br>**预期产出：** CMP 平台上线，覆盖 > 80% 云资源；统一 Dashboard 和告警 |
| 场景二：跨云 DR 方案设计 | **背景：** EnOS 中国区主站部署在阿里云上海，需要 DR 方案<br>**任务：** 设计阿里云上海 → 阿里云深圳的同云 DR + 阿里云 → AWS 新加坡的跨云 DR<br>**挑战：** 跨云数据同步的一致性保障；不同云上的 K8s 版本和配置差异；DNS 切换策略<br>**预期产出：** DR 架构设计、数据同步方案、切换 SOP、演练计划 |
| 场景三：新业务快速上云 | **背景：** 远景储能业务快速扩张，需要在 3 个月内在东南亚上线新的电池管理系统<br>**任务：** 快速评估选云方案（AWS 新加坡 vs 阿里云新加坡国际站），完成基础设施部署<br>**挑战：** 时间紧迫、需复用已有 IaC 模板、需满足新加坡 PDPA 合规<br>**预期产出：** 云环境 3 周内 Ready，应用团队可直接部署 |

## 建议学习路径 {exec}
| Phase 1（入职 1-3 月） | 1. 全面盘点远景各区域的云使用现状（产品、版本、配置、合同）<br>2. 如缺乏阿里云经验，完成阿里云 ACP 认证<br>3. 梳理跨云互联现状和痛点<br>4. 与各区域 IT 团队建立工作关系 |
| Phase 2（4-6 月） | 1. 完成多云策略文档，获得 CIO 批准<br>2. 启动 CMP 选型和 PoC<br>3. 建立跨云 Terraform Module 标准库<br>4. 完成首个跨云互联通道搭建 |
| Phase 3（7-12 月） | 1. CMP 平台上线并推广<br>2. 完成所有区域的跨云互联<br>3. 建立多云成本归集和优化机制<br>4. 沉淀多云最佳实践文档 |

## 成功标准与 KPI {risk}
| 成功标准 | 1. 多云管理平台覆盖 > 80% 云资源<br>2. 新环境交付时间 < 3 周（当前 > 8 周）<br>3. 跨云互联延迟 < 100ms（同区域）<br>4. 云厂商合同年度节省 10%+（通过统一谈判）<br>5. 工作负载在各云间的迁移能力得到验证 |
| 风险 | 1. 阿里云与 AWS/GCP 的产品差异导致统一管理难度大<br>2. 过度追求多云一致性可能丧失各云厂商的差异化优势<br>3. 边缘节点的网络条件限制了统一管理的可行性 |
| 与其他能力域的协同 | **→ 能力一（网络）：** 跨云网络互通是多云管理的物理基础<br>**→ 能力四（IaC）：** 多云 Terraform Module 是 IaC 体系的核心组成<br>**→ 能力五（FinOps）：** 多云成本归集依赖 CMP 平台的数据<br>**→ 能力八（团队）：** 各区域团队需具备多云操作能力 |
