| 对应痛点 | **#2 安全合规差距**（S0-紧急）和 **#7 IAM 管理混乱**（S2-中等）直接相关 |
| 上下游关系 | **上游依赖：** 法务部门提供各国法规解读和合规要求<br>**下游影响：** 安全策略直接影响 #1 网络架构设计（网络分段）、#4 IaC（安全策略即代码）、#6 容灾（数据保护） |
| 能力稀缺度 | **极高** — 同时理解中国等保体系和欧洲 GDPR 的安全架构师极为稀缺 |

## 核心职责详解 {hyp}

### 3.1 — 多国合规策略设计
根据各国法规要求，设计统一的合规框架：明确每个区域的数据驻留、传输、处理规则
✅ | 能够逐国列出合规要求矩阵（数据分类标准、存储位置要求、跨境传输条件、审计要求）；有实际通过等保 2.0 测评和 GDPR 合规审计的项目经验 | → 合规矩阵<br>→ 审计报告
❌ | 对各国法规只有笼统了解，无法具体到技术层面的实施要求；未参与过正式合规审计 | → 合规培训
→ | 聘请各区域合规顾问，协助建立合规知识库 |

### 3.2 — 零信任架构落地
设计和推动零信任安全架构在远景全球基础设施中的落地：身份验证、设备信任、微分段、持续评估
✅ | 能够设计完整的零信任架构方案（BeyondCorp 模型 / NIST 800-207）；有实际零信任项目经验（身份代理、mTLS、ZTNA） | → 零信任方案<br>→ 部署经验
❌ | 安全思维仍停留在边界防御模型（防火墙 + VPN），不了解零信任理念和实施路径 | → 零信任培训
→ | 从 BeyondCorp 模型的 IAP（身份感知代理）开始试点 |

### 3.3 — 数据安全与隐私保护
建立数据分类分级体系、实施数据加密（传输中 + 静态）、数据脱敏、数据生命周期管理
✅ | 能够建立完整的数据分类分级方案（公开/内部/机密/绝密）；熟悉加密方案（KMS、HSM、TDE、Column-level Encryption）；了解数据脱敏技术 | → 数据安全方案<br>→ 加密策略
❌ | 数据安全停留在"加 SSL"层面，不了解数据分类、KMS 管理、数据主权的技术实现 | → 数据安全培训
→ | 从数据资产盘点和分类开始，逐步建立数据安全体系 |

### 3.4 — 安全运营与事件响应
建立安全运营中心（SOC）或利用托管安全服务，实现安全事件的检测、分析、响应和复盘
✅ | 有 SOC 运营经验或 MSSP 管理经验；了解 SIEM（Splunk/ELK/阿里云安骑士）配置和告警规则设计；有安全事件应急响应的实战经验 | → SOC 运营<br>→ IR playbook
❌ | 安全运营靠被动响应，没有系统化的检测和响应流程 | → SOC 培训
→ | 先利用云原生安全产品（Cloud Security Center/GuardDuty/SCC）建立基础检测能力 |

### 3.5 — 供应链与第三方安全
评估和管理第三方服务和供应链的安全风险：云厂商安全评估、SaaS 服务安全审查、开源组件漏洞管理
✅ | 有供应商安全评估体系（安全问卷、SOC2 审计报告审查、渗透测试要求）；了解 SCA（软件组成分析）和 SBOM 管理 | → 供应商评估<br>→ SCA 流程
❌ | 不关注第三方和供应链安全，只关注自有系统 | → 供应链安全
→ | 建立供应商安全评估 checklist，优先评估关键供应商 |

### 3.6 — 安全意识与文化建设
推动全公司的安全意识培训、钓鱼演练、安全编码培训，建立安全文化
✅ | 有安全培训体系建设经验；能设计分层培训方案（全员基础 + 技术团队深度 + 管理层战略）；有钓鱼演练实施经验 | → 培训体系<br>→ 演练报告
❌ | 安全工作仅限于技术团队，不关注全员安全意识 | → 安全培训
→ | 与 HR 协作，将安全培训纳入入职流程和年度考核 |

## 评估维度 — 面试考察要点 {hyp}

### E1 — 多国合规经验
请描述你如何同时满足中国等保 2.0 和欧洲 GDPR 的合规要求
✅ | 能够具体列出两个框架在数据存储、传输、处理方面的差异，并描述如何通过技术架构同时满足（如数据联邦、区域隔离、跨境传输机制选择）；有实际通过审计的经验 | → 合规经验
❌ | 对等保或 GDPR 只有一方面的了解，或者只停留在理论层面，未参与过实际合规项目 | → 需补充
→ | 单一法规深度 + 快速学习能力也可接受，但需在入职 3 月内补齐短板 |

### E2 — 零信任实施
你如何在一个传统 VPN 为主的环境中推进零信任架构？
✅ | 能够描述渐进式零信任路径：从 MFA → 设备信任 → IAP → 微分段 → 持续评估；理解零信任不是推翻一切而是逐步增强；有实际推进案例 | → 零信任能力
❌ | 认为零信任就是"去掉 VPN"或者"加 MFA"，缺乏系统性理解 | → 需培训
→ | 推荐 Google BeyondCorp 白皮书和 NIST 800-207 学习路径 |

### E3 — 安全事件应急
请描述一次你处理的严重安全事件（数据泄露/入侵/勒索等）
✅ | 能够描述完整的 IR 流程：检测 → 遏制 → 消除 → 恢复 → 复盘；事件处理有时间线和量化结果；复盘有改进措施并已落地 | → IR 经验
❌ | 未处理过严重安全事件；或事件处理缺乏系统性，只是"补了漏洞" | → 需演练经验
→ | 安排入职后参与安全应急演练，或者参加 CTF 竞赛提升实战能力 |

## 远景能源适配要求 {exec}
| 中国等保 2.0 | 远景中国区核心系统需满足等保三级：<br>1. 安全物理环境（机房安全）<br>2. 安全通信网络（网络隔离、加密）<br>3. 安全区域边界（入侵检测、访问控制）<br>4. 安全计算环境（身份鉴别、审计）<br>5. 安全管理中心（集中管控）<br>6. 管理制度和人员安全 |
| GDPR 关键要求 | 远景欧洲区（丹麦、德国运营）必须满足：<br>1. 数据处理的合法性基础（同意/合同/正当利益）<br>2. 数据主体权利保障（访问权、删除权、可携权）<br>3. 数据保护影响评估（DPIA）<br>4. 跨境数据传输的合法机制（SCC/BCR）<br>5. 数据保护官（DPO）任命<br>6. 72 小时数据泄露通报义务 |
| 能源行业特殊要求 | 1. SCADA/ICS 系统的安全标准（IEC 62443）<br>2. 关键基础设施保护法规（中国关键信息基础设施保护条例、欧洲 NIS2 指令）<br>3. 风电场远程控制的安全风险（被入侵可能导致物理损害） |
| 跨境数据传输 | EnOS 平台需要跨区域同步部分运营数据（非个人数据），需要设计合规的跨境传输方案：<br>1. 数据分类：区分个人数据 vs 运营数据 vs 设备遥测数据<br>2. 传输机制：SCC（Standard Contractual Clauses）或 BCR（Binding Corporate Rules）<br>3. 技术措施：传输加密 + 访问控制 + 审计日志 |

## 典型工作场景 {exec}
| 场景一：等保三级测评准备 | **背景：** 远景中国区核心系统需通过等保三级年度复测<br>**任务：** 组织等保测评准备，包括技术整改、文档准备、协调测评机构<br>**挑战：** 系统迁移到云上后，等保要求和传统 IDC 有差异（共享责任模型）<br>**预期产出：** 等保测评顺利通过，整改报告，持续合规监控方案 |
| 场景二：GDPR 合规审计 | **背景：** 远景欧洲子公司收到客户的 GDPR 合规审计要求<br>**任务：** 准备审计材料，包括数据处理记录、DPIA 报告、技术安全措施说明<br>**挑战：** 部分数据处理涉及中国总部，需要证明跨境传输的合规性<br>**预期产出：** 审计顺利通过，DPIA 报告更新，跨境传输影响评估 |
| 场景三：安全事件应急响应 | **背景：** 安全监控检测到某区域云环境有异常 API 调用，疑似 Access Key 泄露<br>**任务：** 按 IR 流程快速响应：确认范围、轮换密钥、清理后门、根因分析<br>**挑战：** 多云环境下需要同时在阿里云和 AWS 上排查；跨时区协调<br>**预期产出：** 事件在 4 小时内遏制，根因报告，改进措施落地 |

## 建议学习路径 {exec}
| Phase 1（入职 1-3 月） | 1. 全面了解远景现有安全架构和合规现状<br>2. 与法务团队建立工作关系，获取各国法规 checklist<br>3. 完成 CISSP 或 CCSP 认证（如尚未持有）<br>4. 熟悉各云厂商的安全产品和配置 |
| Phase 2（4-6 月） | 1. 完成多国合规差距分析报告<br>2. 制定安全合规路线图（分区域、分优先级）<br>3. 启动零信任架构试点<br>4. 建立安全事件响应 SOP 和演练机制 |
| Phase 3（7-12 月） | 1. 完成各区域合规整改<br>2. 通过等保三级复测和 GDPR 合规验证<br>3. 零信任架构推广到全部区域<br>4. 建立持续合规监控体系 |

## 成功标准与 KPI {risk}
| 成功标准 | 1. 等保三级测评 100% 通过<br>2. GDPR 合规审计零重大发现<br>3. 安全事件平均响应时间（MTTR）< 4h<br>4. 高危漏洞修复时间 < 72h<br>5. 全员安全培训覆盖率 > 95%<br>6. 零信任架构覆盖 > 50% 关键应用 |
| 风险 | 1. 各国法规持续更新，需要持续跟踪和调整<br>2. 安全合规整改可能影响业务系统的可用性<br>3. 等保测评机构的标准存在主观性差异 |
| 与其他能力域的协同 | **→ 能力一（网络）：** 网络分段和加密策略需在网络架构设计阶段纳入<br>**→ 能力四（IaC）：** 安全策略需要以代码方式管理（Policy as Code）<br>**→ 能力七（可观测性）：** 安全监控需要接入统一可观测性平台 |
