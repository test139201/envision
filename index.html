3. IT/OT 融合 — 为什么远景的基础设施比任何时候都更关键

远景能源的基础设施有一个本质特征，区别于所有纯 IT 企业：它同时承载着 IT 和 OT 两个世界。EnOS 平台不只是一个跑在云上的 SaaS 应用 — 它的下游直接连接着数十万台风机的 PLC 控制器、SCADA 系统和 BMS 电池管理模块。每一个网络架构决策、每一条防火墙规则、每一个 DR 方案，都同时影响着数据安全和物理安全。

更关键的是，远景正处于 IT/OT 融合的加速期。从最初的"采集 → 上云 → 看报表"，正在向"边缘实时推理 → 数字孪生 → 自主控制"演进。这个演进过程对基础设施的要求不是线性增长，而是指数级跃迁 — 远景同时是设备制造商、平台运营商、能源服务商，三重身份叠加意味着基础设施不再只是"保障业务运行"的后台支撑，而是直接影响风机发电效率、电网调度收益、碳资产价值的核心生产要素。

IT/OT 融合的三个阶段 — 远景正处于关键跃迁期

每个阶段对基础设施的要求截然不同。远景正从第二阶段向第三阶段跃迁，这是基础设施压力最大的窗口期。

PHASE 1 · 已基本完成
数据采集与上云

SCADA 采集风机运行数据，通过 IoT 网关上传至云端存储和展示。数据流是单向的（OT → IT），IT 和 OT 物理隔离。

  • 基础设施要求：带宽充足、存储可扩展
  • 网络复杂度：低 — 单向数据管道
  • 安全压力：低 — OT 网络完全隔离
PHASE 2 · 当前阶段
智能优化与闭环控制

云端 AI 分析数据后反向下发控制指令：优化偏航角、调整桨距、预测性维护工单。数据流变为双向（OT ↔ IT）。

  • 基础设施要求：低延迟双向通道、高可用
  • 网络复杂度：高 — IT/OT 必须安全互通
  • 安全压力：高 — 控制指令经过 IT 网络
PHASE 3 · 即将进入
自主运营与实时交易

边缘 AI 在风机端实时决策，数字孪生精确预测，虚拟电厂参与电力市场毫秒级竞价。IT 即 OT，边界消融。

  • 基础设施要求：边缘算力 + 确定性网络 + 实时数据管道
  • 网络复杂度：极高 — 边缘/云/OT 三网融合
  • 安全压力：极高 — 攻击面指数扩大

五个正在发生的融合驱动力

每一个都在对基础设施提出新的、具体的要求

数字孪生 Digital Twin

每台风机都有一个云端数字映射，实时同步转速、振动、温度、功率曲线。用于全生命周期仿真和优化。

基础设施: 风机→边缘→云 延迟 <200ms · 数据完整率 >99.9%
边缘 AI Edge Intelligence

叶片冰冻检测、轴承异常预判需要毫秒级推理。不能等待云端往返 — 模型从云端分发，推理在边缘执行。

基础设施: 每场站 AI 推理节点 · 模型 OTA 分发 · 故障预警提前 72h
虚拟电厂 VPP

将分散的风电、光伏、储能聚合为一个虚拟电厂，参与电力市场竞价。需要跨站点低延迟协调和实时功率调度。

基础设施: 跨站点协调 <500ms · 实时 API 对接电网调度中心
储能智能调度

电池充放电策略基于实时电价、电网需求、电池 SOH 动态优化。BMS 与云端形成毫秒级决策闭环。

基础设施: BMS↔云端 闭环 <100ms · 储能利用率提升 15-25%
碳资产数字化

每度绿电的碳减排量需要精确计量、可追溯、可交易。数据完整性直接影响碳信用价值 — 差的数据 = 打折的碳资产。

基础设施: 数据不可篡改 · 对接碳交易平台 API · 审计追溯链

融合的安全悖论

不融合，远景将丧失数字化竞争优势；过度融合且缺乏安全边界，一次 OT 入侵可能导致风机叶片超速

IT 安全事件 vs OT 安全事件 — 本质差异
IT 系统被攻破OT 系统被攻破
直接后果数据泄露、服务中断风机失控、物理设备损坏、人身安全威胁
影响范围数据和商业声誉设备安全、电网稳定、周边环境
恢复时间小时 ~ 天（重启/恢复）周 ~ 月（物理修复、部件更换）
恢复成本罚款 + 数据修复设备更换 + 停产损失 + 潜在诉讼
环境影响基本无直接环境影响齿轮箱油液泄漏、叶片碎片、有害物质扩散 — 可触发环保监管
级联效应服务降级可隔离至单个微服务融合后系统互联 — 一个节点失效可触发安全系统级联失效
可降级运行可以（返回维护页面、限流）有限（风机只能紧急停机，不能"返回 503"）

核心挑战不是选择"融合"还是"隔离"，而是：在安全的前提下实现最大程度的融合。融合创造业务价值的同时，也创造了新的单点故障 — 原本独立的系统一旦互联，一个节点的失效可能触发级联反应。技术层面：用 Purdue 模型划清边界，用加固的工业 DMZ 严格控制跨区数据流，部署 OT 专用入侵检测与异常检测系统，在 IEC 62443 / NIST SP 800-82 框架内建立纵深防御。管理层面：看不见的资产无法保护 — OT 资产全面可视化（Asset Inventory）是一切安全措施的前提，也是 EY 等咨询机构推荐的融合路线图第一步。

治理与组织 — 融合失败的首要原因不是技术，而是组织

IT 和 OT 团队历史上并行运作，语言不通、目标不同、风险认知截然不同。技术架构再好，如果组织和文化不支撑，融合只会停留在 PPT 上。

IT/OT 融合不是一个抽象概念 — 它对网络、安全、容灾、监控、成本每个域都有具体的、不同于纯 IT 的要求

这个岗位不是一个传统的 IT 基础设施负责人。远景需要的是一个能听懂 OT 工程师说"Modbus 寄存器映射"、同时能和云架构师讨论"Kubernetes Service Mesh"的人。他/她需要在 IEC 62443 的安全框架内，为远景的 IT/OT 融合铺设一条既安全又高效的基础设施底座 — 让每一台风机的遥测数据安全地变成 AI 模型的输入，让每一个云端的优化指令安全地变成风机的执行动作。

5. 痛点诊断 — 远景能源 IaaS 层面十大痛点

基于远景能源全球化运营特点和能源行业特殊性，以下 10 个痛点按严重等级排序。S0 为直接威胁业务连续性的紧急事项，S3 为需长期关注的战略问题。

S0 #1 多地区网络架构碎片化
中国、欧洲、东南亚等各区域网络独立建设，缺乏统一骨干网络。各地 VPN / 专线各自采购，导致 EnOS 平台跨区域数据同步延迟高、链路不稳定，风电场远程监控存在盲区。
业务影响：跨区域 EnOS 数据同步 P99 延迟 > 2s，远程运维响应时效无法保障
S0 #2 安全合规差距
在中国（等保 2.0、数据安全法）、欧盟（GDPR、NIS2）、新加坡（PDPA）等多国运营，数据驻留要求与现有集中式架构冲突。跨境数据传输缺乏合规机制，面临监管处罚风险。
业务影响：合规缺口可能导致欧洲业务受限，最高面临全球营收 4% 的 GDPR 罚款
S0 #3 灾备能力缺失
EnOS 平台和 SCADA 数据采集系统缺乏跨区域灾备方案。单区域故障可能导致风电场监控中断，涉及设备安全和发电效率，对能源行业而言是重大运营风险。
业务影响：区域故障 RTO > 24h，风电场失去实时监控将影响发电收入和设备安全
S1 #4 IaC 覆盖率低
大量基础设施仍为手工配置，各区域运维团队各自维护。无版本控制、无审计轨迹、环境不可复现。变更引发故障时难以快速回滚，且无法实现一致性部署。
业务影响：环境配置漂移导致月均 2-3 次非计划停机，变更回滚平均耗时 4h+
S1 #5 云成本失控
多云环境（阿里云 / AWS / GCP）缺乏统一 FinOps 治理。资源利用率低（部分环境 < 20%）、预留实例过期未续、无 Chargeback 机制。云支出年增长 35%+ 但业务增长未同步。
业务影响：云支出增速远超业务增速，预计年度超预算 20%+
S1 #6 监控可观测性不足
监控工具碎片化：私有云用 Zabbix、AWS 用 CloudWatch、GCP 用 Cloud Monitoring、阿里云用云监控。无统一 Dashboard，无法做跨云关联分析。告警风暴频繁，真正的问题反而被淹没。
业务影响：MTTR > 45 min，告警疲劳导致 P1 事件平均发现延迟 15 min
S2 #7 IAM 管理混乱
各云环境独立的身份体系，未统一至企业 IdP。存在共享账号、部分区域未强制 MFA、特权账号未定期审查。入离职流程未与云权限联动，权限膨胀严重。
业务影响：审计发现多个 dormant 特权账号，安全审计不过关风险高
S2 #8 容量规划缺失
IoT 设备增长（风电机组、光伏面板、储能电池）驱动计算和存储需求不可预测。缺乏容量预测模型，完全被动扩容，多次出现高峰期资源不足导致数据处理积压。
业务影响：季度至少 1 次因容量不足导致 EnOS 数据处理延迟 > 2h
S2 #9 运维团队技能差距
运维团队在传统基础设施领域经验丰富，但云原生技能（K8s、Serverless、IaC、SRE 实践）薄弱。各区域能力参差不齐，中国团队云能力相对较强，欧洲和东南亚团队偏传统。
业务影响：新技术方案落地慢，对云厂商 PS 依赖度高，人效比低于行业水平
S3 #10 供应商锁定风险
中国区深度使用阿里云专有服务（如 DRDS、MaxCompute），海外区域依赖 AWS 专有服务。缺乏抽象层，未来迁移成本高、周期长。当前不紧急，但会限制长期架构灵活性。
业务影响：当前影响可控，但若需迁移预计成本和时间增加 3-5x

6. 依赖关系 — 痛点之间的关键依赖

10 个痛点并非完全独立。以下关键依赖关系决定了执行顺序和并行策略。

#1 网络碎片化 → #3 灾备缺失 跨区域灾备依赖统一骨干网络。#1 的网络改造完成后，#3 的 DR 环境才能跨区域部署和验证。这是最长关键路径。
#4 IaC 框架 → #6 监控可观测性 统一监控平台的部署需要 IaC 框架支撑，确保各区域一致性接入。#4 的 IaC 框架搭建完成后，#6 可以基于此快速部署。
#7 IAM 整改 → #2 安全合规 安全合规验证的前提之一是 IAM 策略的规范化。#7 的 IAM 整改为 #2 的合规审计验证提供基础。两者建议并行推进。

并行策略：10 个痛点 Day 1 全部启动诊断阶段（W1-W8），互不等待。有依赖关系的痛点在执行阶段通过甘特图中的等待段（虚线框）自动同步。无依赖的痛点全程并行推进。
