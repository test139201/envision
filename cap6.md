## 基本信息
| 能力域定位 | 为远景能源的核心系统（EnOS 平台、SCADA 数据采集、企业 ERP/MES）设计和维护跨区域的容灾与高可用架构，确保业务连续性 |
| 对应痛点 | **#3 灾备能力缺失**（S0-紧急）— 核心系统缺乏跨区域 DR，单区域故障 RTO > 24h |
| 上下游关系 | **上游依赖：** 能力一（网络）提供跨区域网络基础设施<br>**下游影响：** DR 能力是安全合规（#2）的关键组成，也是业务连续性承诺（BCP）的技术保障 |
| 能力稀缺度 | **高** — 有能源/工业行业 DR 经验的架构师极少；大多数 DR 经验来自互联网/金融行业 |
| 岗位权重 | 八大能力域中权重 10%，能源行业的 DR 关乎生命安全和财产安全 |

## 核心职责详解 {hyp}

### 6.1 — 业务影响分析（BIA）
对远景全球业务系统进行 BIA 评估，确定每个系统的关键等级（Criticality）、可容忍的最大中断时间（RTO）和数据丢失量（RPO）
✅ | 有完整的 BIA 评估方法论和实操经验；能够与业务部门协作确定系统关键等级分层（P0/P1/P2/P3）；BIA 结果直接指导 DR 方案设计 | → BIA 报告<br>→ 系统分级表
❌ | 对 BIA 概念模糊，无法系统性地评估业务影响；或者只做过 IT 视角的分级，缺乏业务视角 | → BIA 培训
→ | 与业务部门组织 BIA 工作坊，确保分级结果得到业务认可 |

### 6.2 — DR 架构设计（Active-Active / Active-Standby / Pilot Light）
根据 BIA 结果，为不同关键等级的系统设计对应的 DR 架构方案
✅ | 能够对比不同 DR 架构模式的 RTO/RPO/成本权衡：<br>1. Active-Active（RTO≈0，成本最高）<br>2. Warm Standby（RTO 分钟级）<br>3. Pilot Light（RTO 小时级）<br>4. Backup-Restore（RTO 天级）<br>并为不同系统选择最合适的模式 | → DR 方案<br>→ 成本分析
❌ | DR 方案只有一种（要么全 Active-Active 太贵，要么全 Backup-Restore 太慢），缺乏分级设计 | → DR 设计
→ | 从 P0 系统的 DR 方案开始，逐步扩展到 P1/P2 |

### 6.3 — 数据复制与一致性
设计跨区域数据复制方案，确保 DR 站点的数据完整性和一致性
✅ | 精通异步复制 vs 同步复制的选择标准；了解各云的数据复制服务（阿里云 DTS / AWS DMS / GCP Transfer Service）；能够处理数据冲突和最终一致性问题 | → 复制方案<br>→ 一致性策略
❌ | 数据复制依赖手工备份+恢复，没有自动化的持续复制机制 | → 复制建设
→ | 利用云原生复制服务快速建立异步复制，再根据 RPO 要求优化 |

### 6.4 — DR 切换自动化
建立 DR 切换的自动化流程，包括：故障检测 → 决策 → DNS 切换 → 数据库切换 → 应用启动 → 验证
✅ | 有 DR 切换自动化经验；设计过自动化 Runbook/Playbook（Ansible/Terraform）；切换过程有明确的检查点和回滚方案 | → 切换 SOP<br>→ 自动化脚本
❌ | DR 切换完全依赖人工操作，步骤分散在各人脑中，没有标准化流程 | → 自动化建设
→ | 先编写手动 SOP 文档，再逐步自动化关键步骤 |

### 6.5 — DR 演练与持续验证
建立定期 DR 演练机制（至少每季度一次），确保 DR 方案持续有效
✅ | 有 DR 演练组织经验；能够设计不同级别的演练：桌面推演 → 组件切换 → 全量切换；演练结果有复盘和改进措施 | → 演练计划<br>→ 复盘报告
❌ | 从来没有做过 DR 演练，DR 方案只停留在设计阶段 | → 演练规划
→ | 从桌面推演开始（零风险），逐步升级到实际切换演练 |

### 6.6 — 混沌工程
引入混沌工程实践（Chaos Engineering），在生产环境中主动注入故障，验证系统的弹性和 DR 能力
✅ | 了解混沌工程方法论（Principles of Chaos Engineering）；使用过混沌工程工具（Chaos Monkey / LitmusChaos / ChaosBlade）；有在生产环境安全注入故障的经验 | → 混沌工程
❌ | 对混沌工程概念不了解，或者认为"在生产环境注入故障太危险" | → 混沌培训
→ | 从非生产环境开始，建立安全的混沌实验框架，逐步推广到生产 |

## 硬技能要求 {disc}
| BIA 方法论 | ISO 22301 业务连续性管理框架、BIA 评估方法、RTO/RPO 分级标准、业务中断成本计算 |
| DR 架构 | Active-Active（多活）、Active-Standby（主备）、Pilot Light、Backup-Restore 四种模式的设计和实施 |
| 数据复制 | 阿里云 DTS / AWS DMS / GCP Transfer Service；数据库复制（MySQL/PostgreSQL Replication、Redis Sentinel/Cluster）；对象存储跨区域复制 |
| 自动化切换 | Terraform/Ansible 编写 DR 切换 Playbook；DNS 切换（Route53/Alidns/Cloud DNS 权重/故障转移策略）；数据库 Failover 自动化 |
| 容器化 DR | Kubernetes 多集群 DR（Velero/Kasten K10）；StatefulSet 的 DR 策略；PV/PVC 跨区域复制 |
| 混沌工程 | Chaos Monkey / LitmusChaos / ChaosBlade / Gremlin 工具使用；混沌实验设计方法论 |
| 备份管理 | 备份策略设计（3-2-1 规则）、备份验证（定期恢复测试）、备份数据加密和合规 |

## 评估维度 — 面试考察要点 {hyp}

### E1 — DR 实战经验
请描述一次你参与的真实 DR 切换（演练或真实故障）
✅ | 能够描述完整的切换过程：故障检测到决策花了多长时间、切换执行了哪些步骤、遇到了什么意外、最终 RTO 是多少、事后做了哪些改进 | → DR 经验
❌ | 没有参与过 DR 切换，或者参与但说不清楚细节和改进点 | → 经验不足
→ | DR 经验可通过入职后组织演练快速积累 |

### E2 — 能源行业特殊性
能源行业的 DR 与互联网行业有什么本质区别？
✅ | 能够指出：1) 能源系统关乎物理安全（风机失控可能导致人员伤亡）2) SCADA 系统对实时性要求极高 3) 偏远位置的网络可靠性差 4) 监管要求更严格 5) "降级运行"的选项有限（风机不能"返回503"） | → 行业理解
❌ | 用互联网行业的 DR 思维套用到能源行业，不了解 OT 系统的特殊性 | → 需行业培训
→ | 入职后安排到风电场实地参观，理解 OT 系统的运行环境 |

### E3 — 成本与可靠性权衡
如果预算有限，你如何为 10 个系统制定差异化的 DR 策略？
✅ | 能够用 BIA 结果驱动分级决策：P0 系统 Active-Active（不惜成本）、P1 系统 Warm Standby（平衡成本和 RTO）、P2 系统 Pilot Light、P3 系统 Backup-Restore；能给出各级的成本估算 | → 决策能力
❌ | 要么所有系统用同一个 DR 策略（成本爆炸或保护不足），要么无法量化成本和风险 | → 需培训
→ | BIA 工作坊 + DR 架构设计培训 |

## 远景能源适配要求 {exec}
| EnOS 平台 DR | EnOS 是远景最核心的系统：<br>1. 全球部署、多区域实例<br>2. 实时接收数十万 IoT 设备的遥测数据<br>3. 平台中断 → 风电场/储能站失去实时监控 → 安全风险<br>4. 需要 RTO < 4h、RPO < 1h 的 DR 方案 |
| SCADA 系统 DR | SCADA 数据采集系统更加特殊：<br>1. 直接连接风机控制器<br>2. 中断可能导致风机进入安全模式（停机）<br>3. 需要边缘节点的本地缓存和离线运行能力<br>4. 云端 SCADA 系统需要 RTO < 30min |
| 跨云 DR | 远景的多云环境增加了 DR 复杂度：<br>1. 阿里云上海 → 阿里云深圳（同云 DR）<br>2. 阿里云 → AWS 新加坡（跨云 DR，作为终极备份）<br>3. 跨云 DR 涉及数据格式转换、API 差异、网络重建 |
| 合规要求 | 1. 关键信息基础设施保护条例要求核心系统具备容灾能力<br>2. ISO 27001 / ISO 22301 认证需要 DR 演练记录<br>3. 客户合同可能约定 SLA 和 DR 能力 |

## 典型工作场景 {exec}
| 场景一：EnOS 平台 DR 方案设计 | **背景：** EnOS 中国区部署在阿里云上海，需要跨区域 DR<br>**任务：** 设计完整的 DR 方案：数据复制、应用部署、DNS 切换、切换验证<br>**挑战：** 数据量大（PB 级遥测数据）、实时性要求高、复制成本可观<br>**预期产出：** DR 架构设计文档、数据复制方案、切换 SOP、成本估算 |
| 场景二：年度 DR 演练 | **背景：** 年度 DR 演练，需要模拟阿里云上海区域整体不可用<br>**任务：** 组织全量 DR 切换演练，验证 RTO < 4h<br>**挑战：** 需要协调多个团队（平台、应用、网络、安全）同步参与；演练不能影响生产业务<br>**预期产出：** 演练报告（实际 RTO/RPO）、问题清单、改进计划 |
| 场景三：真实区域故障应急 | **背景：** 云厂商通知某区域计划内维护，可能影响部分服务<br>**任务：** 评估影响范围，决定是否预防性切换到 DR 站点<br>**挑战：** 需要快速评估切换的风险 vs 不切换的风险；决策时间有限<br>**预期产出：** 影响评估报告、决策记录、切换/不切换的执行 |

## 建议学习路径 {exec}
| Phase 1（入职 1-3 月） | 1. 完成全部核心系统的 BIA 评估<br>2. 了解远景现有的备份和恢复现状<br>3. 学习 ISO 22301 和阿里云 DR 最佳实践<br>4. 盘点各系统当前的 RTO/RPO 实际水平 |
| Phase 2（4-6 月） | 1. 完成 P0 系统（EnOS、SCADA）的 DR 方案设计<br>2. 启动数据复制通道建设<br>3. 编写 DR 切换 SOP 和自动化脚本<br>4. 组织首次桌面 DR 推演 |
| Phase 3（7-12 月） | 1. 完成 DR 环境建设和验证<br>2. 执行首次全量 DR 切换演练<br>3. 引入混沌工程实践<br>4. 建立 DR 演练季度化机制 |

## 成功标准与 KPI {risk}
| 成功标准 | 1. P0 系统 RTO < 4h（当前 > 24h）<br>2. P0 系统 RPO < 1h<br>3. DR 演练每季度至少 1 次<br>4. DR 切换成功率 > 95%<br>5. 备份恢复测试每月至少 1 次 |
| 风险 | 1. DR 环境建设依赖 #1 网络改造完成（关键路径依赖）<br>2. 跨云 DR 的数据一致性保障技术难度高<br>3. DR 演练可能意外影响生产环境<br>4. DR 环境的持续维护成本需要纳入预算 |
| 与其他能力域的协同 | **→ 能力一（网络）：** DR 站点间的网络连接是 DR 的物理前提<br>**→ 能力三（安全合规）：** DR 方案需满足合规要求（关基保护条例、ISO 22301）<br>**→ 能力四（IaC）：** DR 环境通过 IaC 快速重建和维护<br>**→ 能力七（可观测性）：** DR 站点的监控覆盖和切换告警 |
